# Splitter round 01 ‚Äî Spec 0022 (Berreman Optical Constructor)

## Slicing proposal & rationale

Spec 0022 ("Add Optical Constructor") is a large solution-architecture spec organized into Parts A‚ÄìJ, each a cross-cutting feature area with 8‚Äì12 R-numbered directives plus a per-part acceptance block. The spec's own organization (Parts A‚ÄìJ) is a near-perfect decomposition of *coherent work units*: each Part introduces a bounded set of net-new files, owns a disjoint slice of the acceptance criteria (AC-A* ‚Ä¶ AC-J*), and explicitly cross-references the others by `¬ß`-anchor rather than overlapping them. I therefore slice **one work-slice per Part**, 10 slices total ‚Äî but reordered into dependency order rather than alphabetical order, and I checked each Part really is session-sized (each is one new assembly's worth of files or one analysis layer, a few hours of agent time including build + the three gates).

**Why 10 and not fewer/more:** the Parts are already session-sized and tightly self-contained; merging (e.g. D+E) would bloat a single round past one session and tangle two disjoint AC sets, while splitting a Part would fracture a single `$def`/module owner across rounds and break the fidelity-judge's "every contractful item owned by exactly one slice" expectation. The one judgment call is Part A: it is the load-bearing spine that *names* types (beam tree, project aggregate, units spine) later Parts *fill*. I kept A as one slice but scoped it to "create the projects + define the ¬ßA.4 types + author the serialization seam/schema scaffold + create the test project," explicitly deferring the construction logic (B), units conversions (D), etc., to their owning slices ‚Äî so slice 001 stays buildable and small.

**Dependency-driven ordering (not alphabetical).** Per `splitter.user-md`, core/engine-adjacent work lands before dependents. The order is:

1. **001 ‚Äî Part A** (foundation): five projects + `OpticalConstructor.Tests`, the beam-tree types, the project aggregate skeleton, the JSON serialization seam + schema scaffold, the numerical/units/UI architecture. CRITICAL, lands first.
2. **002 ‚Äî Part D** (materials/units): authors `Units.fs` (the conversion seam) + dispersion catalogue + materials library + import. Moved *before* B/E because both consume `UnitOfMeasure` and the conversion functions.
3. **003 ‚Äî Part B** (construction/beam-tree): fills the ¬ßA.4 types with solver-driving, propagation routing, mirror enforcement, the stack/medium/substrate editors, the default-unit hook, and the beam-tree JSON `$defs`.
4. **004 ‚Äî Part C** (curved elements): extends the beam tree with the lens/curved-mirror zone fan; depends on B's flat-element construction and A's types.
5. **005 ‚Äî Part E** (light sources): `SourceSpec` + editor + expansions; consumes A's `Source` element, D's units seam.
6. **006 ‚Äî Part F** (analysis): pure analysis functions in the Analytics project + the `em.muellerMatrix` completion; consumes E's expanded sources, A's seams.
7. **007 ‚Äî Part G** (optimization): the ALGLIB-backed interface/adapter/fit/synthesis + Wolfram-validation tests; reuses F's `psiDelta` (¬ßF.5) for ellipsometric residuals.
8. **008 ‚Äî Part H** (charts): ScottPlot/Plotly renderers over the engine series + F's colorimetry/field-depth.
9. **009 ‚Äî Part I** (persistence): completes the storage assembly (file IO, validate-on-load, sidecars, autosave, history, export, report); wraps D's `MaterialImport`; provides the `.binz`/undo seams B/F/G referenced.
10. **010 ‚Äî Part J** (UX shell): schematic, repeat builder, templates, drag-and-drop, workspace, environment/prefs, dockable shell, validation, job runner, help/gallery, OpenTK 3D view. The shell that hosts every other surface ‚Äî the natural LAST slice.

This is NOT a rubber-stamp of the spec's section list: it reorders A‚ÜíD‚ÜíB‚ÜíC‚ÜíE‚ÜíF‚ÜíG‚ÜíH‚ÜíI‚ÜíJ for buildability, and it explicitly partitions shared multi-part artefacts (the `BeamTree.fs` type-vs-logic split between 001 and 003; the schema `$defs` filled incrementally across 003/004/005/008 and finalized in 009; the shared `BeamTreeTests.fs`/`ProjectJsonRoundtripTests.fs`/`OptimizationTests.fs` files whose cases are assigned per owning slice).

### Per-slice gate subset & forward-introduced-gate analysis (spec 037 A-1)

The bundle declares three gates. Their runtime support:
- `build` ‚Äî `dotnet build Berreman.slnx`: applies to every slice (all touch `Berreman/**/*.fs`/`.fsproj`/`.slnx`).
- `unit-tests` ‚Äî `dotnet test` over `BerremanTests`: applies to every slice (the `applies_to` globs include `Berreman/Berreman/**`, `Berreman/BerremanTests/**`, and the analytics/domain paths every slice touches).
- `constructor-unit-tests` ‚Äî `dotnet test ‚Ä¶ OpticalConstructor.Tests.fsproj` (spec 0022 ¬ßA.6/P3 testability mandate): its runtime symbol is the `OpticalConstructor.Tests` project, **which slice 001 creates**. Because the test project ships in the *first* slice, the gate is runnable from slice 001 onward ‚Äî there is **no forward-introduced-gate exclusion**: every slice legitimately matches and carries all three gates, and the last slice (010) carries the full set as required. (Had the test project been introduced later, `constructor-unit-tests` would have been dropped from the earlier slices' subsets; it is not, by construction.)

Each slice's "Files modified (planned)" globs match all three gates' `applies_to` patterns, so every slice carries `[build, unit-tests, constructor-unit-tests]`.
