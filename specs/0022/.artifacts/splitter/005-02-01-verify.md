## c. Granularity

- **MAJOR** — **The slicing is under-decomposed: each slice is an entire Part, and several are
  5–10× a single session of work.** The arc-runner runs exactly one worker round per slice and
  requires every enumerated R-N to land in that round (else `commit_ready:false` → escalation →
  failure-budget burn, per `arc-runner.system-md` §"Round protocol" / "What you must NOT do").
  Against that contract these slices are far too large:
  - **001 (Part A):** five new `.fsproj` (Domain/Storage/Optimization/Ui/App) + a test project +
    `Berreman.slnx` wiring + the beam-tree type family + the project aggregate + the JSON
    serialization seam + the schema scaffold — all building green under .NET 10/x64 with
    ScottPlot/Plotly/WebView2/OpenTK/FuncUI in one round.
  - **007 (Part G):** nine new Optimization modules + ALGLIB NuGet integration + a UI page + the
    end-to-end Wolfram validation tests — 11 requirements.
  - **009 (Part I):** eight new Storage modules + schema authoring + four test files + `.gitignore`.
  - **010 (Part J):** ten new UI modules + an environment schema + six test files + drag-drop
    wiring — 12 requirements.

  `splitter.user-md` explicitly warns that slices should be "session-sized — typically 3-8 per big
  spec," that "a 1:1 match with the spec's own internal 'logical pieces' is a smell that the spec
  under-decomposed," and to "lean toward more slices." This decomposition does exactly the 1:1
  Part→slice mapping the guidance flags. The 6-hour `worker_timeout_sec` (see §g) is itself a tell
  that the author expected very long rounds. Recommend splitting the heaviest Parts along their
  natural seams, e.g.: A into (project scaffold + slnx) / (beam-tree types + aggregate + schema
  seam); G into [Core] interface+adapter+design-params+merit+local-refinement / [Standard]
  ellipsometry+inverse-fit+fit-quality+synthesis+page; I into (JSON+schema+file IO+sidecar) /
  (recent/autosave+history+export+report); J into (schematic+repeat+templates+drag-drop) /
  (shell+prefs+validation+jobrunner+help+3D view). This is the headline reason for the verdict —
  it is a rework-the-slicing issue, not a patch.

   and the reason for NEEDS-REVISION. Split at least A, G, I, and J along the seams suggested above;
   the one-round-per-slice arc-runner will otherwise escalate on nearly every slice. Reassign the

   `Avalonia.FuncUI` NuGet is used (only the clone is gated) or that UI views are stubbed without
   FuncUI types until the audit — otherwise the `build` gate is unsound from the first UI slice.
3. **Route slice 009's Part I tests into the gated `OpticalConstructor.Tests` project (§g),** rather
   than a separate `OpticalConstructor.StorageTests`, so AC-I1..AC-I11 actually run under
   `constructor-unit-tests`.
4. **Supply or re-scope the Wolfram reference data for G.11/AC-G9 (§a),** as an explicit fixture
   input or a self-consistency regression with the limitation documented, so slice 007's validation
   test is authorable and non-circular.
5. **(MINOR) Confirm the gate-roster mechanism (§g):** either emit per-slice `.slices/<NNN>.gates`
   snapshot files per the system-prompt contract, or verify the arc-runner build for this project
   reads the manifest `slice_gates:` block as the user-md states. Right-size `worker_timeout_sec`
