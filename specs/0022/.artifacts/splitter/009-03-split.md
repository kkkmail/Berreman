# Splitter round 3

This is a **re-decomposition round**. The round-2 reconciler
(`007-02-01-reconcile.md`) routed to-splitter with four MAJOR and two
MINOR findings, all confirmed (claim-check: 40 CONFIRMED, 0 REFUTED). The
headline finding (§c) is that the prior split was a literal 1:1
Part→slice mapping (ten slices, one per Part A–J), which `splitter.user-md`
flags as an under-decomposition smell against a one-worker-round-per-slice
arc-runner: the largest slices (Part A, G, I, J) bundled whole project
families, 8–10 modules, schema scaffolds, and 10+ requirements each, and
would escalate on nearly every slice.

This round **discards the 1:1 mapping** and re-decomposes into **16
session-sized slices** in build-dependency order, splitting the four
oversized Parts (A→2, G→3, I→2, J→3), additionally splitting Part D
(units-foundation vs materials-catalogue) and Part B (domain-orchestration
vs UI-construction) along the natural build seam, and pulling the JSON
storage core early so every later part can fill its schema `$defs` and
serialize against an existing seam. The four substantive reconciler
findings and both MINOR items are applied (see the final section).

## Slicing rationale (16 slices, build-dependency order)

The parent is a cross-cutting architecture spec: 7 binding constraints
(`## 0`), ten Parts (A–J), 96 acceptance criteria, ~45 new/edited files
across five new F# projects. The unit of work is "a coherent, compilable,
session-sized set of files delivering testable behaviour", not a Part.

**001 — Solution scaffolding & assembly layout** (Part A: §A.1, A.2, A.6,
A.8, A.9).
- *Responsibilities:* the five new F# projects + the gated
  `OpticalConstructor.Tests` host, `Berreman.slnx` wiring (x64), project/
  package references, the engine-consumed-never-forked discipline, the

### Gate-subset reasoning (forward-introduced-gate discipline)

All three bundle gates' runtime support lands in **slice 001**: `build`
and `unit-tests` (the existing `BerremanTests`) are available from the
start, and `constructor-unit-tests` runs the
`OpticalConstructor.Tests` project that slice 001 creates (empty, so its
`count_at_least` baseline starts at 0). No gate depends on a runtime symbol
introduced in a later slice, so **no gate is forward-introduced** and every
slice — including the last — carries the full set. The last slice (016)
carries the full set as required.

## Reconciler feedback applied (round 3)

- **§c MAJOR — under-decomposition (1:1 Part→slice). APPLIED.** Replaced
  the ten 1-Part slices with 16 session-sized slices. The four named
  oversized Parts are split (A→2, G→3, I→2, J→3); Part D and Part B are
  additionally split along the build seam; the JSON storage core is pulled
  early (slice 003) so no part claims "serialize to JSON" before the
  serializer exists. Each slice is a coherent, compilable, testable unit
  of a few files. Verified: every one of the 105 directives and 96
  acceptance criteria is reproduced verbatim and owned by exactly one
  slice; binding constraints appear in all 16.

- **§i MAJOR — FuncUI build path unresolved across UI slices. APPLIED.**
  Slice 001 states the buildable interpretation explicitly: the audit gate
  (constraint 5 / §A.6 / §A.9) applies ONLY to the clone at
  `C:\GitHub\Avalonia.FuncUI.Clone\`; the public MIT `Avalonia.FuncUI`
  NuGet is a distinct artefact and is used so `view` bodies compile and the
  `build` gate is sound from the first UI slice. The clone stays
  unreferenced and the linking mechanism unresolved (AC-A8 / AC-J8). Every
  UI-touching slice (004, 005, 007, 011, 012, 014–016) relies on this and
  reproduces the §A.6/§J.8 verbatim audit text without referencing the
  clone.

- **§g MAJOR — Part I tests in a gate-invisible `StorageTests` project.
  APPLIED.** No `OpticalConstructor.StorageTests` project is created.
  Slice 001 creates the single gated `OpticalConstructor.Tests` host, and
  the parent's `StorageTests/*` test artefacts (RoundTrip, Sidecar,
  History, ExportImport, and the `sample-nk.csv` fixture) are hosted there
  instead (slices 003 and 013), preserving each test's description and ACs
  so AC-I1…AC-I11 actually execute under `constructor-unit-tests`.

- **§a MAJOR — G.11 / AC-G9 Wolfram reference data absent. APPLIED
  (re-scoped, limitation documented).** Slice 011 reproduces §G.11 and
  AC-G9 verbatim (they are contractful), then documents the applied
  fallback in Scope/Risks/Hand-off: because spec §G.0 states the Wolfram
  workflow "lives outside the repository entirely" and no fixture carries
  the recorded values, the worker MUST either (a) carry operator-supplied
  recorded Wolfram values as an explicit fixture under `.manual/`, OR (b)
  implement AC-G9 as a self-consistency/regression test — synthesize a
  target from the real `OpticalSystemSolver`, perturb, and assert
  `LocalRefinement`+`LevenbergMarquardt` recovers it to tolerance over the
  full `DesignParameters→MeritFunction→optimize→AlglibAdapter→
  OpticalSystemSolver` path — and document the no-external-ground-truth
  limitation. Fabricating invented numbers as Wolfram ground truth is
  explicitly forbidden.

- **§g MINOR — emit per-slice `.slices/<NNN>.gates` snapshots. APPLIED.**
  All 16 `.gates` files are written to match the cross-repo snapshot-file
  contract, each mirroring its manifest `slice_gates:` entry.

- **§g MINOR — right-size `worker_timeout_sec` (was 6 h). APPLIED.**
