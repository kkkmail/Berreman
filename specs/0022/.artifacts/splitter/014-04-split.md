## Reconciler feedback applied (round 4)

- **Rec 1 — §b MAJOR: slice 014 forward-references the slice-015 J.9
  validator. APPLIED.** Slice 014's R-2, its `RepeatBuilderTests.fs`
  bullet, the AC-J2 testing-plan line, the Non-requirements note, the
  Risks line, and the Hand-off were all rewritten so that
  `RepeatBuilder.expand` remains the pure
  `List.replicate count cell |> List.concat` with **no dependency on the
  slice-015 `Validation.fs`**, and the `R < 1` rejection assertion is
  owned and tested in slice 015's `ValidationTests.fs` (AC-J9) — not in
  slice 014. The parent §J.2 contract clause ("MUST reject a repeat count
  `< 1` through the J.9 validation seam") is preserved verbatim in R-2;
  only the compile-time cross-slice source reference is removed, which is
  what would have failed the one-round-per-slice `build`/
  `constructor-unit-tests` gates. Slice 015 already tests `R < 1` in
  `ValidationTests.fs`, so no slice-015 change was needed.

- **Rec 2 — §e MINOR: name the Part D by-id material resolver in slice
  004. APPLIED.** Slice 004's R-4 (§D.8) now names, as an explicit
  deliverable, `resolveMaterial : MaterialLibrary -> string -> WaveLength
  -> Result<OpticalProperties, MaterialError>` — a by-id lookup that
  evaluates the entry's `OpticalPropertiesWithDisp` at the supplied
  `WaveLength` via `getEps` (`Dispersion.fs:13`) (a wavelength is required
  because a dispersive entry has no single tensor until one is chosen),
  returning `Result.Error (UnknownMaterialId _)` on an unknown id. The
  files-in-scope `MaterialLibrary.fs` bullet, the `DispersionModelsTests.fs`
  description, the testing plan, and the hand-off were updated to cover and
  test it. This is the function the parent §J.4 ("Part D's
  material-resolution function") and §B.6 ("consume a resolved
  `OpticalProperties`") already delegate to, so naming it is a routing
  decision, not an invention. Consumer slices 005 (§B.6) and 014 (§J.4)
  now point at the named seam in their spec-reference / R-4 /
  Non-requirements text.

- **Rec 3 — §i MINOR: make slice 008's Mueller engine-edit guard
  mandatory. APPLIED.** Slice 008 now mandates a same-slice regression
  guard: a new `Berreman/BerremanTests/MuellerMatrixTests.fs` asserting
  that the completed `member em.muellerMatrix` (`FieldFunctions.fs:93`)
  returns the same matrix as `MuellerMatrix.fromEmFields` (`Fields.fs:636`)
  for a computed solution, registered in `BerremanTests.fsproj` and run by
  the `unit-tests` gate. The Files-modified, testing-plan, Risks, and
  Hand-off sections were updated. The full AC-F3/F5/F7/F8 equivalence
  suite still lands in slice 011's `OptimizationTests.fs` (faithful to the
  parent's test routing at `.spec-md:1002`); only the one engine-core edit
  now ships with its own guard, per the arc-runner's test-first discipline.
