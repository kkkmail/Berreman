# Splitter round 01

Spec **0024** — *Wire the Optical Constructor MVU core into a live FuncUI app
(root host, panel/page views, background jobs, lifecycle, and a launch/render
smoke gate)*. This is round 1; there is no reconciler feedback to apply.

## Slicing proposal

The parent spec is organized U1 → U8 and states: *"It is ordered U1 → U8; U1 is
gating and MUST land first. The splitter SHOULD treat each `## Part U…` heading
as a slice boundary."* Each `## Part U…` is genuinely session-sized (one panel /
page / capability with its own acceptance criteria and a bounded touch-set), so
I keep an **8-slice** plan, one slice per part. This sits at the top of the
project's 3–8-slice band, which is appropriate for a spec that wires eight
distinct operator surfaces onto an already-tested core (the model/`update`/
projection tier is frozen; only the never-authored view tier and two new gates
are in scope).

**One reordering vs. the spec's U-numbering.** U5.1 delivers
`ChartHosts.fs` — the shared `AvaPlot` / `WebView2` / `Canvas` renderer-host seam
that U3, U4, U6, U7, and U8 are each *required to reuse* ("U3/U4/U6/U8 MUST reuse
these adapters rather than embed their own hosting", §U5.1; U3.2 / U4.2 / U6.1 /
U7.4 / U8.4 all host through it). Because the arc-runner implements slices in
manifest order, a shared dependency must land before its consumers — and the
project's own ordering rule (`splitter.user-md`: foundations land before
dependents) mandates exactly this. The spec's only *binding* ordering constraint
is "U1 first"; the rest may be topologically ordered. So I sequence **Part U5 as
slice 002**, right after the U1 foundation and before its consumers. U2 (stack
editing) depends only on U1, so it follows at slice 003. This is the single
deviation from the literal U1→U8 reading order; every contractful item is
preserved unchanged.

#### Slice 001 — Root MVU host + shell skeleton + ui-smoke gate (Part U1)

- **Responsibilities.** Add the Elmish host packages to both projects; author the
  root `RootModel`/`RootMsg`/`init`/`update`/`view` in a new `Shell.fs`; mount the
  Elmish program from `MainWindow` replacing the static `Program.fs:39` mount;
  rewrite `AppShell.shellView`/`panelView` to switch on panel id; render the
  `stack` panel read-only to prove the loop; stand up the headless
  `OpticalConstructor.Ui.Tests` project, the smoke test, and the `ui-smoke` /
  `ui-tests` gates. Native-host adapters degrade gracefully headlessly (§U1.8).
- **Natural boundary.** `dotnet build` green with the two new package refs and the
  new test project in the solution (AC-U1.4); the `ui-smoke` gate exits 0 with
  every panel/page rendering one frame headlessly (AC-U1.3); the `stack` panel
  shows the node's layer rows (AC-U1.2) and Construction is the active landing
  page (AC-U1.1).
- **Touch-set.** `Shell.fs` (new), `ConstructionView.fs` (new, read path),
  `AppShell.fs`, `OpticalConstructor.Ui.fsproj`, `Program.fs`,
  `OpticalConstructor.App.fsproj`, `OpticalConstructor.Ui.Tests.fsproj` (new) +
  `*.fs` (new), `Berreman.slnx`, `.gates/OpticalConstructor/ui-smoke.gates`
  (new), `.gates/OpticalConstructor/ui-tests.gates` (new).
- **Risk — high.** Foundational and gating: every later slice depends on the
  `Shell.fs` surface and the `ui-smoke` gate. Worries: version-aligning
  `Avalonia.FuncUI.Elmish` / `Elmish` against the pinned FuncUI 1.6.0 / Avalonia
  11.3.4; keeping the headless Avalonia dependency out of the pure
  `OpticalConstructor.Tests` project so `constructor-unit-tests` stays
  Avalonia-free; rendering all five panels one frame headlessly without throwing.

#### Slice 002 — Renderer-host adapters + chart panel (Part U5)

- **Responsibilities.** Add `ChartHosts.fs` (AvaPlot / WebView2 / Canvas
  adapters, the reusable seam); render 1-D ScottPlot charts in the AvaPlot host,
  the 3-D Plotly surface in the WebView2 host; wire chart-settings + on-plot
  cursor readouts; add the `Chart` `RootMsg` case.
- **Natural boundary.** The `chart` panel renders `Plot1DView` in an `AvaPlot`
  and `Refresh()`es on model change (AC-U5.1); settings reach the plot via
  `applyToScottPlot`/`applyToPlotly` and cursors update `Readout.Markers`
  (AC-U5.2); the panel renders one frame headlessly.
- **Touch-set.** `ChartHosts.fs` (new), `ChartView.fs` (new), `Shell.fs`,
  `OpticalConstructor.Ui.fsproj`, UI-test `*.fs`.
- **Risk — medium.** Native renderer hosting (WebView2/AvaPlot) must degrade to
  an "unavailable" placeholder under headless / missing-runtime (§U1.8) while the
  control instance lives in the host layer (§0.5).

#### Slice 003 — Stack editing + async node-solve (Part U2)

- **Responsibilities.** Make the `stack` panel editable (dispatch every
  `StackMsg` as `Construction (EditStack …)` through the frozen
  `ConstructionPage.update`); attach the async node-solve `Cmd` running
  `solveSubtree` off-thread and dispatching `NodeSolved` marshaled per §0.4;
  surface per-node busy, delete-confirm, and single-level undo UI.
- **Natural boundary.** A stack edit marks the node busy, re-solves off-thread,
  and refreshes with no manual reload (AC-U2.1); delete-with-children confirms
  same-row and removes with undo available (AC-U2.2).
- **Touch-set.** `ConstructionView.fs` (edit), `Shell.fs`, UI-test `*.fs`.
- **Risk — medium.** Off-UI-thread re-solve with correct `Dispatcher.UIThread.Post`
  marshaling at per-node granularity.

#### Slice 004 — Materials panel (Part U3)

- **Responsibilities.** Browse `MaterialLibrary` with `byCategory`/`byNameContains`

- **Natural boundary.** Dispersion preview renders in WebView2 / placeholder

- **Risk — low–medium.** WebView2-hosted Plotly preview degrading gracefully and

#### Slice 007 — Synthesis / Fit page + background jobs (Part U7)

- **Responsibilities.** Open `SynthesisFitPage.Model` on demand into the root

### Per-slice gate subsets (and forward-introduced-gate analysis)

The spec bundle gate set is
`[build, unit-tests, constructor-unit-tests, impl-log-structure,
state-of-world-structure, ui-smoke, ui-tests]`. I assign the **full set to every
slice**, which is correct here because:

- §0.7 mandates that *every* slice ends `build`-green, `constructor-unit-tests`-green,
  `ui-smoke`-green, and `ui-tests`-green (once the UI test project exists).
- The structural gates (`impl-log-structure`, `state-of-world-structure`) and the
  core `unit-tests` (BerremanTests) gate apply to every slice — their `applies_to`
  globs (`Berreman/**/*.fs`, `Berreman/**/*.fsproj`, etc.) match every slice's
  touch-set, and `unit-tests` is `count_at_least` so an unchanged core passes
  trivially.
- **Forward-introduced gates (spec 037 A-1).** `ui-smoke` and `ui-tests` depend on
  the headless `OpticalConstructor.Ui.Tests` project (and the two new gate files),
  whose runtime ships in **slice 001 (Part U1)**. Slice 001 is the *first* slice,
  so there is no earlier slice from which to omit them: the slice that ships the
  runtime — and every later slice — carries both gates. No forward-introduced gate
  needs to be dropped from any earlier slice, because all the gating
  infrastructure is front-loaded into U1 (which is exactly why the spec makes U1
  "gating and land-first"). The two gate files already exist on disk
  (`.gates/OpticalConstructor/ui-smoke.gates`, `ui-tests.gates`) from the gates
  cycle, matching this assignment.

The last slice (008) carries the full set, as required.
