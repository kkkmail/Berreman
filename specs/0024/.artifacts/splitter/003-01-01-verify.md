## i. Other

- **MINOR -- F# compile-order / `RootMsg` forward-reference footgun (recurs every
  slice).** F# compiles files in `.fsproj` order; a file may only reference
  symbols defined earlier. `Shell.fs` holds `RootMsg` *and* `Shell.view`, and
  `Shell.view` calls the leaf `*View.fs` modules -- so the leaf views must be
  registered *before* `Shell.fs`. But the slices also say each leaf view
  "dispatches as `Construction (EditStack …)` / `Source …` / `Chart …`", i.e.
  emits `RootMsg`-constructor values, which would force the leaf view to come
  *after* `Shell.fs` -- a cycle. The idiomatic resolution (leaf view takes a
  sub-`Msg` dispatcher, e.g. `dispatch : ConstructionPage.Msg -> unit`, and
  `Shell.view` wraps it as `(Construction >> dispatch)`; or a tiny shared
  `RootMsg` types module precedes the leaf views) is obvious and keeps the spec's
  end-to-end dispatch intact, but it is not called out anywhere. Because slice
  001 establishes the pattern every later slice copies, and because a wrong
  choice breaks the `build` gate (the first gate), the worker on 001 should be
  steered toward the sub-`Msg`-dispatcher shape and toward inserting each new
  `*View.fs`/`ChartHosts.fs` *before* `Shell.fs` in `OpticalConstructor.Ui.fsproj`
  (current order has `AppShell.fs` at line 56; `Shell.fs` must land after the
  leaf views it calls). Non-blocking; the slicing itself is sound.

- **MINOR -- gate-source mechanism: manifest `slice_gates:` block vs. sibling
  `.slices/<NNN>.gates` snapshots.** The cross-repo `arc-runner.system-md`
  (step 5) and `verifier.system-md` (Phase 1.3) describe per-slice gates living
  in a sibling `<slice-stem>.gates` snapshot file ("the manifest header carries
  no `gates_file:` key anymore"). This arc has **no** `.slices/*.gates` snapshot
  files; instead the manifest carries an inline `slice_gates:` YAML block, and
  the project's `arc-runner.user-md` explicitly says "the arc-runner reads each
  slice's gate set from the manifest's `slice_gates:` block." So the gate
  assignment is present, complete, and matches the bundle gates and each slice's
  testing plan -- functionally nothing is missing and no gate is unresolved.
  Flagging only so the reconciler is aware of the mechanism drift: if the running
  arc-runner build expects sibling `.gates` snapshots rather than the manifest
  block, the splitter would need to materialise the per-slice snapshots. Given
  the project prompt is explicit and recent, the manifest block is treated as
  authoritative here. Non-blocking.
