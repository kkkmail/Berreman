## Findings

### F1: Hand-rolled JSON object walk instead of the shared `ProjectJson.options` stack

- **Worker added:** `Localization.parse` (Localization.fs:101-122) and `loadFromFile` (Localization.fs:127-132) — a bespoke `JsonDocument.Parse` + `EnumerateObject()` walk that builds `Map<string, Map<string,string>>` element by element, wrapped in its own `try ... with e -> Error (ResourceParseError ...)`.
- **Existing helper:** `ProjectJson.options`, the project's single shared `JsonSerializerOptions` (System.Text.Json + FSharp.SystemTextJson), at `Berreman/OpticalConstructor/OpticalConstructor.Storage/ProjectJson.fs:70-80`. It is documented verbatim as "The single shared `JsonSerializerOptions` (§I.1 — registered once)" and already round-trips F# `Map<_,_>` values (proven by the `Map` round-trip assertions in `ProjectJsonRoundtripTests.fs` and consumed by `UserEnvironment.deserialize`, UserEnvironment.fs:277). `strings.json`'s `{ key: { "en": …, "ru": … } }` shape *is* `Map<string, Map<string,string>>`, so `JsonSerializer.Deserialize<Map<string, Map<string,string>>>(text, ProjectJson.options)` parses the whole resource in one call.
- **Why it matters:** This introduces a *second* JSON-reading mechanism in a codebase whose binding rule (ProjectJson.fs:11-16) and this slice's own constraints (0.7 / I.5) both insist on one reused JSON stack. The worker's state-of-the-world even claims that reuse ("One JSON stack, reused", lines 77-80) — but only for the fieldless `Language` DU; the resource *body* is parsed by a divergent hand-walk that the shared options would cover for free. The hand-walk additionally *skips* non-object / non-string entries silently (Localization.fs:111-117), which diverges from every other load path in the repo (`deserialize`, `deserializeProject`), each of which returns an `Error` on malformation rather than dropping data. Two parse idioms with different malformation semantics is the maintenance cost: a future change to the resource format lands in code that does not share the project serializer settings or its error discipline.
- **Suggested action:** Deserialize into `Map<string, Map<string,string>>` via the shared `ProjectJson.options` (the Ui project already references Storage, and `ProjectJson.options` is pure, so the loader stays headless-testable), reusing the same `try Ok … with e -> Error (ResourceParseError e.Message)` shape already present. If the skip-bad-entries leniency is a deliberate requirement (a half-translated file must still load its valid keys), keep the hand-walk but state in a comment that it intentionally does NOT reuse the shared stack — as written, the divergence reads as oversight rather than choice. Advisory; the judge decides.

### F2: Parallel `LocalizationError` DU over `StorageError`'s missing/parse territory

- **Worker added:** `LocalizationError` with `ResourceMissing of path` and `ResourceParseError of message` (Localization.fs:67-69).
- **Existing helper:** `OpticalConstructor.Storage.Errors.StorageError` (Errors.fs:19-25) — the repo's established "surfaced, never thrown" IO error channel, carrying `JsonParseError of string`, `SchemaMissing`, and `FileIoError of exn`, i.e. exactly the three failure modes the loader hits (parse failure, missing file, read failure). The Result-returning, never-throw contract the worker re-implements at Localization.fs:127-132 is the very discipline `StorageError` was built for ("NEVER throws across its own boundary", Errors.fs:4-6).
- **Why it matters:** It is a second, small error vocabulary for "file missing / file unparseable" that a reader now has to learn alongside `StorageError`. This is the weaker finding: the shape match is only partial (`ResourceMissing` carries a path that `SchemaMissing` lacks), and `StorageError` also carries storage-only cases (`BinzError`, `SidecarMissing`) that would be noise in a UI string loader.
- **Suggested action:** Defensible to leave as-is — a focused two-case type avoids importing storage-layer semantics into the UI loader, consistent with the worker's deliberate choice to keep `Localization.fs` dependency-light. If kept, a one-line note on why a separate channel (rather than `StorageError`) is the right boundary would settle it. Low confidence; the judge decides.

## Bottom line

One substantive near-miss (F1: the resource body is parsed by a bespoke `JsonDocument` walk with skip-on-malformation semantics instead of the project's single shared `ProjectJson.options` stack, against the codebase's explicit "one JSON stack" rule) and one weaker, defensible near-miss (F2: a small parallel error DU over `StorageError`'s territory). Neither is a correctness defect and each has a plausible design rationale, so my read is that these do not compel a re-spawn — F1 warrants a documented decision (reuse the shared stack, or say why the lenient hand-walk is intentional) and F2 is fine to leave. The judge owns the routing.
