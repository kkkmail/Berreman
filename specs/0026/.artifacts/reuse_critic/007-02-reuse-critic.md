## Findings

### F1: `GroupsLibrary.serialize`/`deserialize` re-implement the `ProjectJson` envelope+validate+bind seam

- **Worker added:** `GroupsLibrary.serialize` (`OpticalConstructor.Storage/GroupsLibrary.fs:73-79`) and `GroupsLibrary.deserialize` (`:84-97`).
- **Existing helper:** `ProjectJson.serializeProject` / `ProjectJson.deserializeProject`, at `OpticalConstructor.Storage/ProjectJson.fs:85-91` and `:98-111` — and a *second* copy already in `UserEnvironment.serialize`/`deserialize` (`OpticalConstructor.Ui/UserEnvironment.fs:300-324`).
- **Why it matters:** the two new functions are structurally identical to `ProjectJson`'s. `serialize` is the same five steps — `SerializeToNode(value, ProjectJson.options)` → `AsObject()` → set `schemaVersion` → `ToJsonString(options)`, wrapped in `try/with → JsonParseError`. `deserialize` is the same ladder — `try Ok(JsonDocument.Parse json)` → `use doc = doc` → `match validate doc.RootElement` → `doc.RootElement.Deserialize<'T>(ProjectJson.options)`. Only the bound type, the `validate` callee, and the `schemaVersion` literal differ. With this slice the envelope round-trip now lives in **three** modules; an envelope-format change or a parse-error-handling fix must be made in three places and the third can silently drift. The worker *did* correctly reuse `ProjectJson.options` (the shared converter stack the spec names, §I.1 / R-4) — this finding is about the thin serialize/deserialize wrapper around it, not the options.
- **Suggested action:** extract a generic pair in the Storage layer beside `ProjectJson` — e.g. `serializeWithEnvelope options schemaVersion value` and `deserializeValidated options validate json` — and route `ProjectJson`, `UserEnvironment`, and `GroupsLibrary` through it. Advisory; the judge decides whether the third copy is the moment to hoist (it touches files outside this slice).

### F2: `GroupsLibrary.validate` + its lazy loader copy the `SchemaValidation` scaffolding (only `collectMessages` was actually shared)

- **Worker added:** `GroupsLibrary.validate` (`GroupsLibrary.fs:58-68`) plus the schema-locate-and-load block `schemaRelativePath`/`schemaFullPath`/lazy `schema` (`:34-50`).
- **Existing helper:** `SchemaValidation.validate` (`OpticalConstructor.Storage/SchemaValidation.fs:64-76`) and its own `schemaRelativePath`/`schemaFullPath`/lazy `schema` (`:17-36`) — same Storage layer, directly reachable. (`UserEnvironment.fs:271-295` is a third copy of the same block.)
- **Why it matters:** credit where due — attempt 03 made `SchemaValidation.collectMessages` `public` (`SchemaValidation.fs:45`) and `GroupsLibrary.validate` now calls it (`GroupsLibrary.fs:66`), so the nested-`results.Details` detail is genuinely shared and the spec's named seam (G.3.1 / R-4, *"reuse the schema-validate-on-load seam, `SchemaValidation.fs:61`"*) is honoured at the message-flattening layer. What remains copied is everything *around* that one call: the `EvaluationOptions(OutputFormat = OutputFormat.List)` → `schema.Evaluate(element, options)` → `if results.IsValid` → empty-fallback arm is verbatim, and the whole `if File.Exists … FromText … FileIoError / SchemaMissing` lazy loader is verbatim with only the schema file name changed. The obstacle is that `SchemaValidation.validate` and its loader are hard-wired to the single project `$id` — that is precisely the seam that should have been parameterized rather than copied a second time.
- **Suggested action:** add `SchemaValidation.validateAgainst : JsonSchema -> JsonElement -> Result<unit,StorageError>` (the evaluate/IsValid/collectMessages body) and `SchemaValidation.loadSchema : string -> Result<JsonSchema,StorageError> Lazy` (the relative-path loader), then have both `SchemaValidation.validate` and `GroupsLibrary.validate` call them with their own `$id`. That makes the groups path reuse the *whole* seam, not just its tail. Advisory.

### F3: a fourth copy of the per-user app-data folder path

- **Worker added:** the app-data folder block in `GroupsLibrary.libraryPath` (`GroupsLibrary.fs:104-109`): `Path.Combine(GetFolderPath ApplicationData, "Softellect", "Berreman", "OpticalConstructor")`.
- **Existing helper:** the identical literal already appears in `RecentFiles.prefsPath` (`OpticalConstructor.Storage/RecentFiles.fs:28-33`, **same layer**), `Report.fs:271-273` (**same layer**), and `UserEnvironment.settingsPath` (`OpticalConstructor.Ui/UserEnvironment.fs:333-338`, Ui layer).
- **Why it matters:** the `Softellect/Berreman/OpticalConstructor` app-data root is now hard-coded at four sites; relocating the vendor/app folder is a four-edit change with no single source. The SoW/worker comment defends the copy on layering grounds ("the Storage layer does not reference the Ui layer where the environment store lives") — fair for the Ui copy, but it misses the closer precedent: `RecentFiles` and `Report` are in the *same* Storage assembly, so a shared root was reachable without crossing a layer. (`RecentFiles.prefsPath` is `private`, so it could not be reused as written — but that argues for hoisting it, not for adding a fourth private copy.)
- **Suggested action:** extract one `appDataFolder ()` in the Storage layer and have `GroupsLibrary`, `RecentFiles`, and `Report` reuse it (and `UserEnvironment` too, since Ui→Storage is the allowed direction). Low severity; reasonable to defer given the pre-existing duplication. Judge decides.

### F4: `confirmGateOverlay` re-creates the scrim + centred-dialog chrome that `modalOverlay` already owns

- **Worker added:** `confirmGateOverlay` (`OpticalConstructor.Ui/Ribbon.fs`, slice-007 overlay block) — an outer scrim `Border` (`SolidColorBrush(Color.FromArgb(120uy,0uy,0uy,0uy))`) wrapping a centred white / gray-bordered `Border` with `padding 16.0`.

- **Why it matters:** the overlay chrome is duplicated verbatim. `modalOverlay`'s fixed single-close-button footer genuinely does not fit a two-button Confirm/Cancel gate (which is why the worker forked it), but the *scrim + centred dialog shell* is the reusable part and was copied rather than shared — a future restyle of the modal backdrop must now be applied twice. (The worker *did* reuse `Controls.destructiveGate`, `Controls.toggle`, `Controls.defaultButton`/`disabledButton`, and `modalOverlay` itself for the element-dialog and context-menu overlays — good reuse; this is only the confirm-gate backdrop.)
- **Suggested action:** extract `overlayShell (child : IView) : IView` (the scrim + centred `Border`) and have both `modalOverlay` and `confirmGateOverlay` wrap their content with it. Advisory; small.

## Bottom line

The diff faithfully reuses the genuinely risky targets — the engine, `History.EditHistory`, `ProjectJson.options`, the now-`public` `SchemaValidation.collectMessages`, `deleteNow`, and the `Controls`/`modalOverlay` UI helpers — so attempt 03 closed the cycle-1 "dropped nested-error detail" gap, and what remains (F1–F4) is the envelope / evaluate-scaffolding / app-data-path / overlay-chrome boilerplate *around* those reused cores. That residue is duplication the repo already tolerates (`UserEnvironment` is a second JSON-store copy; the app-data path is four-way), so the worker followed precedent rather than inventing fresh. My read: F1 and F2 are the strongest because both cited helpers sit in the *same* Storage layer and the spec explicitly asked the groups store to reuse that seam, making the third copy a fair trigger to finally extract — but none is a correctness defect and an extraction reaches beyond this slice, so this sits at the borderline for forcing a re-spawn. The judge holds that call.
