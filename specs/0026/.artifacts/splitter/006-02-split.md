## §g — the gate-roster finding, and how it was fixed

The reconciler traced the live tool's source and proved:

- `run_arc_runner.py:2085` calls `load_slice_gates(slice_path)`.
- `load_slice_gates.py:43-62` reads the **sibling `<slice-stem>.gates` snapshot
  file**; when it is absent it returns `[]`, and the arc-runner then WARNs
  (`run_arc_runner.py:2086-2091`) and runs the slice with **zero gates**.
- `load_manifest.py:201` only *strips* the manifest `slice_gates:` block
  (`strip_slice_gates_block`); it never parses it for gate selection.
  `parse_slice_gates_block` runs only in the splitter's own self-validation
  (`run_splitter.py:1379`), never on the arc-runner path.

So the manifest `slice_gates:` block round 1 authored is real and correct, but it
is **documentation only** for the runtime — the arc-runner reads gate rosters from
`.slices/<NNN>.gates`. Round 1 emitted none, so every slice would have run with no
gates (build / unit-tests / constructor-unit-tests / ui-smoke / ui-tests /
impl-log-structure / state-of-world-structure all silently skipped), committing
every slice with no acceptance verification.

**Fix:** I emitted `.slices/001.gates` … `.slices/007.gates`, one per slice, each
a multi-document YAML gate-descriptor stream copied **verbatim** from the resolved
spec gates (`.spec-gates`). Each file carries exactly the gate subset already
declared for that slice in the manifest's `slice_gates:` block (no re-derivation,
no divergence) — the manifest block now serves as the human-readable mirror of the
runtime snapshots, exactly as the reconciler directed ("mirroring the per-slice
rosters already in the manifest's `slice_gates:` block, which can stay as
documentation").

The snapshot files use the same shape as the reference arc `specs/0024/.slices/
*.gates` (which the live arc-runner has already consumed), and all seven were
parse-validated and field-validated through the live loader
(`ai_strategy_generator.spec_writer.gates.descriptor.read`) before this summary
was written.
