# Splitter round 01

## Slicing proposal — Spec 0026 (Optical Constructor UI)

Spec 0026 is a large, multi‑part feature (Parts A–K plus operator‑facing UX
commitments) that adds a top‑down Optical Constructor screen to the existing
Avalonia + FuncUI desktop app under
`Berreman/OpticalConstructor/`. The spec is internally organised by priority
parts; those parts are *how the author explained the work*, not the work
plan. Several parts are too large for one session (Part E + the MVU surface),
a few are tightly coupled (D needs E's registry; the Part C view interactions
need the MVU page), and a couple are cross‑cutting (K touches every action;
I underpins every later UI string). I therefore decompose into **7 work
slices**, keeping the two CRITICAL "land first" parts (A, B) as their own
slices and ordering domain before UI per the project's slicing rubric.

Verified before slicing: every spec `file:line` anchor resolves in the
working tree; both test projects the gates target already exist
(`OpticalConstructor.Tests`, `OpticalConstructor.Ui.Tests`), so the
`constructor-unit-tests`, `ui-smoke`, and `ui-tests` gate runtimes predate
this arc — **no gate is forward‑introduced** (see gate analysis below).

---

### Slice 001 — Element geometry, placement & rotation domain model (CRITICAL — land first)

- **Responsibilities:** Part A. New pure `Placement.fs` domain layer:
  `ElementPlacement` record, `BoundingBox` + N1/N2 normals, R1/R2/R3 with
  per‑axis locks (R3 locked by default), `CatalogueKind` DU +
  `toConstructorElement`, value‑id slot, emission smart constructor, default
  box proportions. Extends `Project.fs` + project schema with per‑element
  placement.
- **Natural boundary:** `PlacementTests.fs` proves rest‑pose normals, default
  locks, `r2+R2` additivity, R3 non‑orthogonality, the emission invariant,
  catalogue mapping, and a placement round‑trip + schema validation
  (AC‑A1…A7).
- **Expected file touch‑set:** `Placement.fs` (new), `Project.fs` (edit),
  `optical-constructor-project.schema.json` (edit), `PlacementTests.fs` (new).
- **Estimated risk:** Medium — extending the permissive `constructorElement`
  `$def` (schema `:201`) without renaming the nine reserved anchors
  (Project.fs:42) and keeping round‑trip green; R3 non‑orthogonality math.

### Slice 002 — Ray model, snapping & schematic tracing (CRITICAL — land first)

- **Responsibilities:** Part B. New pure `RayModel.fs`: CR + 8 side rays,
  IRG/RRG/TRG, snapping + distance‑preserving downstream travel, orphaning
  rules, default LS/D placement (2.0 m apart) + LS reset, schematic Snell
  tracing (n=1.0/1.5). No Avalonia type.
- **Natural boundary:** `RayModelTests.fs` proves the 45° eight‑ray circle,
  mirror RRG‑only, distance‑preserving travel, default placement + reset,
  orphaning, Snell bending (AC‑B1…B6).
- **Expected file touch‑set:** `RayModel.fs` (new), `RayModelTests.fs` (new).
- **Estimated risk:** Medium — the downstream‑travel transform and Snell
  geometry are exact headless assertions; floating‑point tolerance must
  follow `MatrixComparison.fs`.

### Slice 003 — Localization: English and Russian

- **Responsibilities:** Part I. `Localization.fs` runtime loader +
  completeness check, `strings.json` (EN/RU, build‑copied/runtime‑read),
  `language` on `EnvironmentSettings` + env schema, startup completeness
  check + copyable error in `Program.fs`. Sequenced before the big UI slices
  so later UI text resolves through it.
- **Natural boundary:** `LocalizationTests.fs` proves runtime load,
  missing‑string fallback + copyable error, no‑translate symbol set; the
  `language` round‑trips via `EnvironmentRoundTripTests` (AC‑I1…I4).
- **Expected file touch‑set:** `Localization.fs` (new), `strings.json` (new),
  `UserEnvironment.fs` (edit — `language`), env schema (edit — `language`),
  `Program.fs` (edit — startup check), `LocalizationTests.fs` (new), fsproj
  content‑item entry for `strings.json`.
- **Estimated risk:** Medium — `strings.json` build‑copy + base‑directory
  read must mirror the schema content‑item pattern (SchemaValidation.fs:17);
  copyable error must not crash the headless host.

### Slice 004 — Optical table, drawer & top‑down rendering geometry; standardized controls

- **Responsibilities:** Part C (table + drawer + rendering geometry) + Part J
  (controls). `Table.fs` (plate + view state, default 1.2×2.0×0.10 m),
  `Drawer.fs` (one standard cylinder drawer keyed by kind + value id, source/
  detector shadings, show/hide box), `ConstructorTable.fs` (top‑down layout +
  named drawing‑weight constants), `Controls.fs` (flavours). Extends
  `Project.fs` + schema with the table.
- **Natural boundary:** `ConstructorViewTests.fs` proves drawer geometry +
  shadings + show‑box, active‑element indicator threshold, CR‑only default,
  control‑flavour usage (AC‑C1, C4, C5, C6, J1).
- **Expected file touch‑set:** `Table.fs`, `Drawer.fs`, `ConstructorTable.fs`,
  `Controls.fs`, `ConstructorViewTests.fs` (new); `Project.fs`, project schema
  (edit — table).
- **Estimated risk:** Medium — extend `Project.fs` + schema with the table
  while preserving slice‑001 placement anchors; assert the WCAG indicator
  threshold from pure values; keep `Schematic.fs` untouched.

### Slice 005 — Centralized commands, mouse/keyboard interaction & the constructor MVU surface

- **Responsibilities:** Part E + the FuncUI `Canvas` MVU page + the Part C
  view interactions that depend on it. `Commands.fs` single registry
  (three‑state bindings), `ConstructorView.fs` (model/msg/update/view +
  interaction handlers reading the registry), pan/zoom/rotate/reset, table
  selection, rotations, slide/move, drag‑to‑place, global commands, key‑map
  persistence on env settings.
- **Natural boundary:** `CommandRegistryTests.fs` proves single‑source
  bindings, rotation step + lock inertness, slide/move/plain‑drag + redraw,
  drag‑to‑place snap, undo/redo/save/cancel bindings, zoom + reset‑view, table
  selection (AC‑E1…E5, AC‑C2, AC‑C3).
- **Expected file touch‑set:** `Commands.fs`, `ConstructorView.fs`,
  `CommandRegistryTests.fs` (new); `UserEnvironment.fs`, env schema (edit —
  key map).
- **Estimated risk:** High — largest UI slice; the registry must be the sole
  source for both key map and mouse map (AC‑E1 / constraint 0.4); headless
  simulation of wheel/drag gestures is the test care‑point.

### Slice 006 — Ribbon‑and‑menu shell, front door, Legacy entry; element catalogue & value‑id modal

- **Responsibilities:** Part D + Part F. `Ribbon.fs` (tabs + collapsed menus
  generated from the slice‑005 registry; contextual Element tab; Legacy
  entry), Shell `Page` extension + default landing (constructor as front
  door), catalogue surfacing (LS/LP/CP/S/Lens/Flat/Curved/D, no analyzer),
  value‑id empty modal, `LocalHelp.fs`.
- **Natural boundary:** ribbon/menu/page view tests prove tab set from the
  registry, ribbon↔menu single‑source equivalence, contextual tab, Legacy
  entry + default landing, catalogue contents, value‑id modal (AC‑D1…D4,
  AC‑F1, F2).
- **Expected file touch‑set:** `Ribbon.fs`, `LocalHelp.fs` (new); `Shell.fs`,
  `Program.fs` (edit — default landing), `ConstructorView.fs` (edit — modal +
  contextual tab), Ui.Tests (extend).
- **Estimated risk:** High — rerouting the app's front door through the
  existing `RootModel`/`RootMsg` without breaking the legacy `Page` paths the
  ui gates already cover; ribbon↔menu equivalence is the constraint‑0.4 proof.

### Slice 007 — Element groups, experiment collections, multiple detectors & undo/redo participation (LAST)

- **Responsibilities:** Part G + Part H + Part K. `Groups.fs` (`ElementGroup`,
  member modes, `ExperimentCollection`), `GroupsLibrary.fs` + new groups
  schema (separate per‑user app‑data JSON), multiple detectors + primary,
  and — now that all project‑mutating actions exist — wiring every action to
  `push` `EditHistory` snapshots (multi‑level) plus destructive‑action safety
  gates.
- **Natural boundary:** `GroupsRoundTripTests.fs` + undo/detector view tests
  prove member modes + lossless swap, primary detector, separate‑JSON
  round‑trip + schema, multi‑level undo across all actions, destructive‑action
  confirm/undo (AC‑G1…G3, AC‑H1, AC‑K1, K2).
- **Expected file touch‑set:** `Groups.fs`, `GroupsLibrary.fs`, groups schema,
  `GroupsRoundTripTests.fs` (new); `Commands.fs`, `ConstructorView.fs`,
  `Ribbon.fs`, optionally `Project.fs`/schema (edit), Ui.Tests (extend).
- **Estimated risk:** High — Part K is cross‑cutting (every earlier action
  must `push` without double‑pushing); lossless swap; the separate app‑data
  groups file load/save/validate; as the final slice any earlier regression
  fails the full gate set.

## Per‑slice gate subsets and forward‑introduced‑gate analysis

Bundle gate set: `[build, unit-tests, constructor-unit-tests, ui-smoke,
ui-tests, impl-log-structure, state-of-world-structure]`.

Assignment rationale (by "Files modified (planned)" globs):
- `build`, `unit-tests`, `impl-log-structure`, `state-of-world-structure` —
  every slice (every slice touches `Berreman/**/*.fs`; the structural checks
  are universal).
- `constructor-unit-tests` (runs `OpticalConstructor.Tests`, which references
  Domain/Storage/Optimization, **not** Ui) — slices touching Domain, Storage,
  or `OpticalConstructor.Tests`: 001, 002, 003 (via `UserEnvironment` ↔
  `EnvironmentRoundTripTests`), 004, 005 (same), 007. **Omitted on 006**,
  which is a pure Ui/App slice that `OpticalConstructor.Tests` does not cover.
- `ui-smoke`, `ui-tests` (run `OpticalConstructor.Ui.Tests`) — slices touching
  `OpticalConstructor.Ui` / `.App` / `.Ui.Tests`: 003, 004, 005, 006, 007.
  **Omitted on 001 and 002**, which are domain‑only (no Ui file).
- **Last slice (007) carries the full set**, as mandated.

**Forward‑introduced gates (spec 037 A‑1 / spec 0110 class):** none. I checked
each gate's `command` symbol against what each slice ships. `build`,
`unit-tests`, `constructor-unit-tests`, `ui-smoke`, and `ui-tests` all target
projects that already exist in the tree (`Berreman.slnx`, `BerremanTests`,
`OpticalConstructor.Tests`, `OpticalConstructor.Ui.Tests` with its existing
`SmokeTests.fs` + per‑panel tests). No gate's runtime support is first
introduced by a later slice, so no earlier slice has to omit a gate on the
forward‑introduction axis — the omissions above are purely glob‑scope based.
New per‑part tests (`PlacementTests`, `ConstructorViewTests`,
`CommandRegistryTests`, `GroupsRoundTripTests`, …) ship in the same slice as
the production code they exercise, so each gate stays green at every slice.
